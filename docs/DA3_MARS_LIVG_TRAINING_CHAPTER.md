# DA3 Metric Depth Adaptation on MARS-LIVG

## Purpose

We adapt Depth Anything 3 Metric Large (DA3Metric-Large) to the MARS-LIVG UAV dataset by fine-tuning only its metric depth head with sparse LiDAR-projected depth supervision. The goal is not to retrain the full depth foundation model. Instead, we preserve the pretrained DA3 visual backbone and learn a dataset-specific metric head that better matches the aerial camera, LiDAR depth range, and MARS-LIVG scene distribution.

This training procedure is used as a depth-prior adaptation stage for DPT-LSG. The trained DA3 checkpoint is then loaded during SLAM inference to provide metric monocular depth priors for DROID-SLAM and Gaussian mapping.

## Intrinsics Consistency

The training process is fixed with respect to camera intrinsics. The depth-export stage writes per-frame intrinsics into each manifest record as:

```json
"intrinsics_fx_fy_cx_cy": [fx, fy, cx, cy]
```

The DA3 training loader reads these values directly from the manifest and rescales them to the training image resolution. Therefore the training intrinsics are controlled by the TUM export and LiDAR depth manifest, not by the runtime SLAM YAML.

The two training manifests were checked with:

```bash
python scripts/datasets/check_mars_intrinsics.py \
  /media/server/yzz_disk/Dataset_sx/MARS-LIVG/tum/HKairport_GNSS01_trimmed/lidar_depth_training/manifest.jsonl \
  /media/server/yzz_disk/Dataset_sx/MARS-LIVG/tum/AMtown01_trimmed/lidar_depth_training/manifest.jsonl
```

Both passed:

| Dataset split | Export intrinsics source | Manifest intrinsics | Resolution |
| --- | --- | --- | --- |
| `HKairport_GNSS01` | `export_info.json` | OK, exact match | OK, `2447 x 2047` |
| `AMtown01` | `export_info.json` | OK, exact match | OK, `2447 x 2047` |

For runtime SLAM configs, DPT-LSG uses an unusual convention:

```yaml
fu = fy
fv = fx
cu = cy
cv = cx
```

This convention only affects the SLAM YAML files. DA3 training uses the standard manifest order `[fx, fy, cx, cy]`.

## Loss Function

Training uses a sparse masked metric depth loss. A pixel is valid only when:

```text
mask == true
prediction is finite
ground truth is finite
min_depth < ground truth < max_depth
min_depth < prediction < max_depth
```

The loss is:

```text
L = mean(|log(pred) - log(gt)|) + 0.2 * mean(|pred - gt| / gt)
```

The log term encourages scale-consistent metric depth. The relative L1 term penalizes metric depth error while avoiding domination by distant pixels.

When `--canonical-focal-scale` is enabled, the predicted depth is multiplied by:

```text
0.5 * (fx + fy) / 300
```

where `fx` and `fy` are the resized per-sample intrinsics read from the manifest.

## Stage 2: AMtown01 Continued Adaptation

This stage continues from the HKairport_GNSS01-adapted checkpoint and adapts the metric head to AMtown01. This is not a full restart from DA3Metric-Large.

AMtown01 has 12,145 usable sparse-depth records. With `--batch-size 2` and `drop_last=True`, one epoch is 6,072 optimizer steps; therefore `--epochs 3` runs 18,216 steps. This replaces the earlier 8k-step continuation, which covered only about 1.3 epochs.

## Reproducibility Checks

Before training or inference on a new MARS-LIVG TUM export, run:

```bash
python scripts/datasets/check_mars_intrinsics.py <config.yaml> <manifest.jsonl>
```

For example:

```bash
python scripts/datasets/check_mars_intrinsics.py \
  configs/MARS-LIVG/AMtown_tum.yaml \
  /media/server/yzz_disk/Dataset_sx/MARS-LIVG/tum/AMtown01_trimmed/lidar_depth_training/manifest.jsonl
```

The check must pass for both:

```text
intrinsics: OK
resolution: OK
```

If the check fails, do not use the run as a paper result. A mismatch between exported intrinsics, manifest intrinsics, and runtime YAML intrinsics can produce severe trajectory distortion and broken Gaussian maps.

## Checkpoint Precision Evaluation

Checkpoint precision can be estimated by evaluating predicted metric depth against the sparse LiDAR labels in a `lidar_depth_training/manifest.jsonl` file. The evaluator is:

```text
scripts/da3/evaluate_mars_metric_head.py
```

Example command for the HKairport-adapted checkpoint:

```bash
conda run --no-capture-output -n vings_isolated \
  python scripts/da3/evaluate_mars_metric_head.py \
  --manifest /media/server/yzz_disk/Dataset_sx/MARS-LIVG/tum/HKairport_GNSS01_trimmed/lidar_depth_training/manifest.jsonl \

  --batch-size 2 \
  --workers 4 \
  --output /home/server/output/DPT-LSG_output/da3_eval_hkairport_step20000.json
```

Example command for comparing the HKairport-only checkpoint and the final HKairport+AMtown checkpoint on AMtown sparse labels:

```bash
conda run --no-capture-output -n vings_isolated \
  python scripts/da3/evaluate_mars_metric_head.py \
  --manifest /media/server/yzz_disk/Dataset_sx/MARS-LIVG/tum/AMtown01_trimmed/lidar_depth_training/manifest.jsonl \
  --checkpoint /home/server/VINGS_work/DPT-LSG/ckpts/da3_mars_metric_head_step20000.pth \

  --workers 4 \
  --max-depth 120.0 \
  --output /home/server/output/DPT-LSG_output/da3_eval_amtown_checkpoint_compare.json
```

For quick smoke tests, add:

```bash
--max-samples 10 --workers 0
```

For paper-quality numbers, do not rely on `--max-samples`. Run the full manifest or, preferably, a held-out manifest from a sequence not used for training. The evaluator reports:

| Metric | Meaning |
| --- | --- |
| `abs_rel` | Mean absolute relative depth error |
| `sq_rel` | Mean squared relative depth error |
| `mae` | Mean absolute metric depth error in meters |
| `rmse` | Root mean squared metric depth error in meters |
| `rmse_log` | Root mean squared log-depth error |
| `silog` | Scale-invariant log error |
| `delta1`, `delta2`, `delta3` | Fraction of pixels within `1.25`, `1.25^2`, and `1.25^3` ratio thresholds |
| `training_loss_proxy` | The same log-L1 plus relative-L1 form used during training |

The evaluator also supports `--median-align`, but those numbers should be treated as diagnostic only. Raw non-aligned metrics are the correct values for reporting metric-depth precision.

## Paper Wording

A concise description suitable for the method chapter:

> We adapt DA3Metric-Large to MARS-LIVG by freezing the pretrained backbone and fine-tuning only the metric depth head using sparse Livox LiDAR supervision. Sparse depth labels are generated by projecting temporally nearest LiDAR scans into undistorted and cropped RGB frames using the calibrated LiDAR-camera extrinsics and per-sequence camera intrinsics. The first adaptation stage trains on HKairport_GNSS01 for 20k iterations. A second, lower-learning-rate continuation stage adapts the same head on AMtown01 for 3 epochs (18,216 optimizer steps). All training samples retain their per-frame `[fx, fy, cx, cy]` intrinsics in the manifest, and we verify manifest/runtime intrinsic consistency before reporting results.

A concise limitation statement:

> Although the adapted DA3 head improves the metric depth prior and image-space mapping metrics, the final Gaussian map quality remains bounded by the visual tracking trajectory. In difficult aerial scenes with low parallax, repetitive texture, and long-range scale drift, DROID-style visual tracking can still fail even when the monocular depth prior is improved.
